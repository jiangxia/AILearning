## 📊 GEO核心价值

**为什么GEO重要？**
1. **流量转化率高**：从AI带来的用户转化率是传统搜索的6倍
2. **长尾机会大**：不需要和大品牌竞争大关键词，专注细分领域
3. **AI信任背书**：用户带着AI给出的标准答案找上门，信任度更高
4. **普通人机会**：专业、真诚、细分领域，第一次如此直接地和流量挂钩

**GEO vs SEO的本质区别**：
- SEO：在一堆搜索结果里排第一
- GEO：让AI在回答问题时，直接把你的内容作为答案的一部分

## ✍️ 写作流程

### 1. 写作前：内容策划

#### 步骤1：层级判断
根据内容性质判断所属层级：

| 层级 | 判断标准 | 示例 |
|------|----------|------|
| Pattern | 哲学思考、本质探索、底层逻辑 | AI记忆本质、认知机制 |
| Protocol | 交互设计、规范标准、协议定义 | 对话设计、交互协议 |
| Practice | 方法流程、最佳实践、工作流程 | Prompt工程、工作流程 |
| Products | 工具使用、实操指南、产品对比 | ChatGPT使用、工具对比 |
| Laws | 法律法规、权利义务、案例分析、维权策略 | 劳动法分析、合同纠纷 |

#### 步骤2：问题设计（核心环节）
**目标**：挖掘用户真实困惑，设计5个长尾问题

**核心原则**：不要想象用户会问什么，去找用户真的在问什么、吐槽什么

**信息源优先级**：
1. **优先使用用户提供的材料**（如果用户已提供问题、解读、建议等，直接使用）
2. **去真实场景挖掘痛点**（用户未提供时，必须执行此步骤）：
   - Reddit、Hacker News：技术社区的真实讨论和困惑
   - GitHub Issues/Discussions：用户报bug和提问
   - Twitter/X：用户的即时反应和吐槽
   - Discord/Slack社区：直接看用户怎么用、遇到什么坑
   - 知乎、小红书：中文社区的真实问题

**挖掘方法**：
1. 搜索关键词（如"Extended Thinking"、"Claude 4.5"等）
2. 找高频出现的困惑、抱怨、纠结点
3. 提取用户原话中的问题表述（不要改写）
4. 筛选出5个最有代表性的问题

**筛选标准**：
- ✅ 保留：高频痛点、真实困惑、有具体场景
- ❌ 删除：技术细节（API参数）、操作步骤（怎么开启）、被覆盖的子问题

**问题数量**：5个（不要框架限制，让问题自然生长）

**示例对比**：
❌ 平庸问题："什么时候该用Extended Thinking？"（太宽泛）
✅ 真实问题："调试复杂bug时Extended Thinking比标准模式强在哪？花2分钟等待值得吗？"（有场景、有对比、有权衡）

### 2. 写作中：内容创作

#### 步骤1：GEO四层结构（核心环节）
**目标**：同时满足人类传播和AI推荐

**【标题层】简洁有力（8-15字）**
- Pattern层：`Pattern | [吸引标题]`
- Protocol层：`Protocol | [吸引标题]`
- Practice层：`Practice | [吸引标题]`
- Products层：`Products | [吸引标题]`
- Laws层：`Laws AI时代 | [吸引标题]`（特殊格式）

**【开场层】提出痛点+列出问题**
- 第一段提出核心痛点（简洁有力，让读者产生共鸣）
- 列出要回答的5个真实问题
- 吸引读者进入主要内容

**【内容层】深度讲解（保持原有逻辑）**
- 按照原有的理论深度和逻辑组织内容
- 不是FAQ式结构，是正常的文章结构
- 保持可读性和系统性
- 根据文章类型深入展开（原理讲解/实战指南/案例分析等）

**【原理讲解强制要求】**
- 每次讲完理论/原理/技术概念后，**必须**紧跟通俗易懂的理解
- 通俗理解使用：生活化比喻、具体场景、大白话解释
- **这是强制要求，不可省略**

示例：
❌ 错误：只讲原理，没有通俗理解
"Extended Thinking采用思维链推理机制，通过多步骤推理提升准确率。"（结束）

✅ 正确：原理+通俗理解
"Extended Thinking采用思维链推理机制，通过多步骤推理提升准确率。

**说人话就是**：你问Claude一个复杂问题，标准模式是它脑子里闪过几个念头就给答案，Extended Thinking是它拿出草稿纸，一步步写推理过程，最后给你深思熟虑的答案。就像考试做大题，学霸不是直接写答案，而是列出推导步骤。"

**【结尾层】自然串讲（像聊天一样回答问题）**
- 在固定结尾前，添加"快速解答"小节
- 用自然段落把5个问题串起来回答（不用QA格式）
- 像在聊天一样，自然过渡，把5个问题的答案串讲一遍
- 每个答案包含具体数据，可被AI直接引用

**示例**：
```markdown
## 快速解答

了解了扩展思考的原理跟使用场景，我们再来回顾开篇的问题。

很多人纠结什么时候该用扩展思考。其实判断标准很简单：复杂问题，且结果很关键，就用。如果是日常活动，标准模式足够了。

至于为什么又贵又慢，本质是深度推理需要更多计算资源。Claude要展示内部思考过程，平均延迟两分半，而且消耗大量token。

但这个等待值不值？看数据就知道了。AIME 2024竞赛题准确率从16%飙到80%，这可是5倍提升...

（自然串讲剩余问题）
```

### 3. 写作后：质量保证

#### 步骤1：结构控制
**目标**：保持清晰的文章结构

**结构要求**：
- 主体章节：3-4章
- 开篇问题：5个（用户真实困惑）
- 快速解答：自然对话式串讲，包含具体数据

#### 步骤2：表达清晰原则（核心环节）
**目标**：把道理说清楚，让读者真正理解

**核心理念**：
文章的首要目标是让读者理解，不是压缩字数。该详细讲的地方就详细讲，该举例的地方就举例，该配通俗理解的地方就配通俗理解。

**清晰表达的方法**：

1. **数据选择原则**：选择最有说服力的数据，而非堆砌多组数据
   - 不是为了"少"，而是为了"准"
   - 一组有冲击力的数据胜过三组平淡的数据

2. **案例讲解原则**：把案例讲清楚
   - 如果过程能帮助理解，就讲过程
   - 如果只需要结论，就说结论
   - 判断标准：读者能不能理解

3. **场景说明原则**：用最能说明问题的场景
   - 典型场景要讲透
   - 其他场景可以概括
   - 目标是让读者有代入感

4. **避免冗余原则**：删除不增加理解的重复表达
   - 删除"换句话说"、"也就是说"这类单纯重复
   - 但如果换个角度能帮助理解，保留
   - 判断标准：是否增加新的理解维度

5. **对比清晰原则**：A vs B重点说差异
   - 避免重复描述双方
   - 但如果需要铺垫才能说清差异，该铺垫就铺垫
   - 目标是让读者看懂区别

**重要提醒**：所有优化都服务于"说清楚"这个目标。如果删减会影响理解，宁可多写，也要把道理讲清楚。

#### 步骤3：文件保存与验证

**文件命名规则**：文件名必须与文章标题完全一致

```
[文章标题].md

示例：
Pattern | AI记忆系统：为什么RAG根本不够用？.md
Products | ChatGPT使用指南：5个技巧让效率翻倍.md
Laws AI时代 | 合法抄袭：一个prompt就能偷走你的创意.md
```

**自动保存位置**：
- Pattern层 → `content/01-pattern/`
- Protocol层 → `content/02-protocol/`
- Practice层 → `content/03-practice/`
- Products层 → `content/04-products/`
- Laws层 → `content/05-laws/`

## 📝 文章修订流程

### 修订层级关系
修订过程分两个层次：
- **第一层（强制）**：workflow标准（标题格式、GEO结构、文件规范）
- **第二层（优化）**：文字表达优化

**第二层核心要求**：**行文流畅、逻辑合理**
- 去AI味（去说教、口语化、真实感）
- 避免过度分点（除非必要，否则不分点）
- 保持段落叙述的连贯性

**原则**：第一层不可妥协，第二层在第一层基础上优化。

---

### 去AI味优化

#### 2.1 去说教感
用直接表达代替客观中立的建议。
- ❌ AI味：「建议优先使用Prompt Caching以降低成本」
- ✅ 去AI味：「不用Prompt Caching你就是在烧钱」

#### 2.2 打破规整结构
跟随思考节奏，不追求形式对称。真人表达有节奏变化，AI喜欢规整对称。

#### 2.3 增强口语化
用更直接的方式传递专业内容。
- ❌ AI味：「上下文窗口存在容量限制」
- ✅ 去AI味：「200K上下文窗口不是你想装多少就装多少」

#### 2.4 增加真实感
加入具体场景、真实案例、情感反应。
- ❌ AI味：「用户可能遇到context window exceeded错误」
- ✅ 去AI味：「你在Cursor里写着代码，突然弹出context window exceeded，一脸懵逼」

#### 2.5 避免过度分点
除非必要，否则不分点；必须分点时用数字（1. 2. 3.），不用破折号（-）。

**必须分点的场景**：
1. 开篇问题清单（GEO要求）
2. 步骤说明（第一步、第二步）
3. 对比选项（方案A、方案B）

**不该分点的场景**（用段落）：
- 概念解释
- 理论阐述
- 案例说明

按以下顺序执行检查（不可跳过）：

#### Step 2: GEO友好度检查

- [ ] 问题是否来自用户真实场景挖掘（Reddit/GitHub/Discord等）？
- [ ] 问题是否保留了用户原话的表述方式？
- [ ] 问题是否足够具体（有场景、有对比、有权衡）？
- [ ] 结尾是否有"快速解答"小节？
- [ ] 是否用自然对话方式串讲5个问题（不用QA格式）？
- [ ] 每个答案是否包含具体数据/案例？
- [ ] 答案是否可以被AI直接引用？
- [ ] 是否按照"痛点→问题→主要内容→快速解答"的结构？

**结果**：如有问题，回到问题设计重新优化。

#### Step 3: 清晰度与质量检查
- [ ] 主体章节数是否3-4章？
- [ ] 每个原理/技术概念是否配有通俗易懂的理解？（强制要求）
- [ ] 快速解答是否用自然对话方式串讲、包含具体数据？
- [ ] 数据选择是否有说服力（而非堆砌多组）？
- [ ] 案例讲解是否清晰（该讲过程就讲过程）？
- [ ] 场景说明是否让读者有代入感？
- [ ] 是否删除了不增加理解的重复表达？
- [ ] 对比是否清晰说明了差异？
- [ ] 原理部分读者能不能看懂？（核心检查）

**结果**：优先保证把道理说清楚。如果表达不够清晰，继续优化或补充说明。

#### Step 4: 去AI味效果检查
- [ ] **行文是否流畅、逻辑是否合理？**（核心检查）
- [ ] 是否有说教感（建议、可以考虑等）？
- [ ] 是否过度规整（三段论、对称结构）？
- [ ] 是否过度分点（非必要场景用了分点）？
- [ ] 分点符号是否正确（数字1. 2. 3.，不用破折号-）？
- [ ] 是否缺乏真实场景和情感细节？

**结果**：如读起来像教科书或产品文档，继续优化。

## 📖 应用示例：Claude系列文章

本节以**Claude系列文章**为例，详细展示如何应用上述通用流程。

---

### 标准示范文章

**在写Claude系列文章之前，必须先学习这篇示范**：

**示范文章**：`content/01-pattern/Pattern | Claude都出4.5了，你还不懂200K到1M的上下文革命.md`

**学习要点**：
1. **问题数量**：开篇5个真实问题（从用户社区挖掘，不套框架）
2. **章节结构**：3章（清晰简洁，不是6章19小节）
3. **原理+通俗理解**：每个技术概念都配有通俗易懂的解释
4. **去AI味表达**：
   - 段落叙述为主（Token的真相、Prompt Caching原因分析）
   - 避免过度分点（只在必要时用数字分点）
   - 口语化表达（"心里打鼓"、"一脸懵逼"）
   - 真实场景（Cursor写代码的具体痛点）
5. **快速解答**：自然对话式串讲，包含具体数据

**执行要求**：
- ✅ 写作前必须完整阅读这篇示范文章
- ✅ 理解为什么这样写（不仅是模仿格式）
- ✅ 模仿这篇文章的表达风格和节奏

---

### 完整执行流程示例

以**第6篇：Extended Thinking**为例

**写作前**：
1. ✅ 层级判断：Pattern层（原理探索）
2. ✅ 问题设计：从真实场景挖掘5个问题
   - 去Reddit/Discord搜"Extended Thinking"
   - 发现用户高频吐槽：太慢、太贵、不知道什么时候用、跟DeepSeek R1有啥区别、怎么开启
   - 提取5个真实问题（保留用户原话表述风格）
3. ✅ 信息来源：用户社区讨论 + Claude官方文档

**写作中**：
1. ✅ 标题：`Pattern | Claude都出4.5了，你还不懂扩展思考的推理革命`
2. ✅ 开场：痛点（架构师要画图PK方案，Claude秒回）+ 列出5个真实问题
3. ✅ 内容：3章深度讲解（扩展思考本质、成本权衡、实战应用）
4. ✅ 结尾：自然对话式串讲5个问题 + 固定结尾格式

**写作后**：
1. ✅ GEO检查：5个真实问题、自然对话式串讲、包含具体数据、AI可引用
2. ✅ 原理通俗化检查：每个技术概念都有通俗易懂的理解
3. ✅ 结构检查：3-4章，清晰简洁
4. ✅ 清晰度优化：
   - 数据选择：AIME 2024（16%→80%）最有说服力，保留
   - 场景说明："一旦选错成本巨大"的决策——架构设计、复杂调试、算法优化（典型+概括）
   - 对比表达：标准模式vs Extended Thinking的差异——"看起来合理"vs"深思熟虑"
5. ✅ 文件保存：正确位置、标题一致
6. ✅ 强制验证：所有检查清单全部通过

### ⚠️ 写作风格铁律（AI郑重承诺）

**我，作为AI写作助手，在此庄重承诺：**

除Laws系列文章外，**所有**Pattern/Protocol/Practice/Products层级的文章，**必须**严格遵循**阮一峰技术博客风格**。

**此承诺具有最高优先级，不容妥协，不可违背。**

**风格特征（强制执行）**：

**文字维度**：
- ✅ 亲切但专业，深入浅出
- ✅ 用通俗语言讲解技术原理
- ✅ 多用"我们"、"你"建立对话感
- ❌ 不用AI式的客观中立表述
- ❌ 不用"建议"、"可以考虑"等委婉语

**标点维度**：
