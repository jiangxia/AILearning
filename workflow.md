## 📊 GEO核心价值

**为什么GEO重要？**
1. **流量转化率高**：从AI带来的用户转化率是传统搜索的6倍
2. **长尾机会大**：不需要和大品牌竞争大关键词，专注细分领域
3. **AI信任背书**：用户带着AI给出的标准答案找上门，信任度更高
4. **普通人机会**：专业、真诚、细分领域，第一次如此直接地和流量挂钩

**GEO vs SEO的本质区别**：
- SEO：在一堆搜索结果里排第一
- GEO：让AI在回答问题时，直接把你的内容作为答案的一部分

## ✍️ 写作流程

### 1. 写作前：内容策划

#### 步骤1：层级判断
根据内容性质判断所属层级：

| 层级 | 判断标准 | 示例 |
|------|----------|------|
| Pattern | 哲学思考、本质探索、底层逻辑 | AI记忆本质、认知机制 |
| Protocol | 交互设计、规范标准、协议定义 | 对话设计、交互协议 |
| Practice | 方法流程、最佳实践、工作流程 | Prompt工程、工作流程 |
| Products | 工具使用、实操指南、产品对比 | ChatGPT使用、工具对比 |
| Laws | 法律法规、权利义务、案例分析、维权策略 | 劳动法分析、合同纠纷 |

#### 步骤2：GEO问题挖掘（核心环节）
**目标**：找到用户真实会问的长尾问题

**优先级原则**：
1. **优先使用用户提供的材料**（如果用户已提供问题、解读、建议等，直接使用）
2. 如果用户未提供，再执行以下操作

**操作**（用户未提供材料时）：
1. 从权威信息源和用户反馈平台搜索关键词
2. 提取5-7个高频具体问题（建议20词以上的长尾问题）
3. 验证信息真实性和权威性
4. 按场景分类（创业者/开发者/产品经理/企业决策者/法律从业者等）

**信息源选择原则**：
- 根据文章领域，优先选择权威、可靠的信息源
- 用户讨论仅作为问题发现渠道，结论必须有权威依据
- 法律类文章必须基于法律法规、司法判例等官方来源

**长尾问题特征**：
- 具体场景（不是泛泛而谈）
- 真实痛点（用户实际遇到的问题）
- 足够细节（20词以上）

### 2. 写作中：内容创作

#### 步骤1：GEO四层结构（核心环节）
**目标**：同时满足人类传播和AI推荐

**【标题层】简洁有力（8-15字）**
- Pattern层：`Pattern | [吸引标题]`
- Protocol层：`Protocol | [吸引标题]`
- Practice层：`Practice | [吸引标题]`
- Products层：`Products | [吸引标题]`
- Laws层：`Laws AI时代 | [吸引标题]`（特殊格式）

**【开场层】提出痛点+列出问题**
- 第一段提出核心痛点（50-80字左右）
- 列出要回答的5-7个长尾问题（让AI知道文章覆盖哪些问题）
- 吸引读者进入主要内容

**【内容层】深度讲解（保持原有逻辑）**
- 按照原有的理论深度和逻辑组织内容
- 不是FAQ式结构，是正常的文章结构
- 保持可读性和系统性
- 根据文章类型深入展开（原理讲解/实战指南/案例分析等）

**【结尾层】快速解答（提供标准答案）**
- 在固定结尾前，添加"快速解答"小节
- 简要回答开篇列出的5-7个问题
- 每个答案提供可引用的"标准答案"（包含具体数据）
- 格式：**Q: 问题？** A: 答案。

### 3. 写作后：质量保证

#### 步骤1：GEO友好度检查（核心环节）
**目标**：确保文章可被AI引用

**检查清单**：
- [ ] 开篇是否列出了5-7个长尾问题？
- [ ] 文章结尾是否有"快速解答"小节？
- [ ] 每个问题的答案是否包含具体数据/案例？
- [ ] 答案是否可以被AI直接引用（标准答案格式）？
- [ ] 文章是否保持了理论深度和可读性？
- [ ] 是否按照"痛点→问题→主要内容→快速解答"的结构？

#### 步骤2：文件保存与验证

**文件命名规则**：文件名必须与文章标题完全一致

```
[文章标题].md

示例：
Pattern | AI记忆系统：为什么RAG根本不够用？.md
Products | ChatGPT使用指南：5个技巧让效率翻倍.md

```

**自动保存位置**：
- Pattern层 → `content/01-pattern/`
- Protocol层 → `content/02-protocol/`
- Practice层 → `content/03-practice/`
- Products层 → `content/04-products/`
- Laws层 → `content/05-laws/`

**强制验证清单**：
- [ ] 文件位置是否正确（根据层级保存到对应目录）
- [ ] 文件名是否与标题完全一致
- [ ] 标题格式是否符合规范（Laws层特殊格式）
- [ ] Laws层文章是否包含法律声明
- [ ] 是否包含完整的固定结尾内容
- [ ] GEO检查清单是否全部通过

**错误修正流程**：
- 发现文件位置错误 → 立即移动到正确目录
- 发现标题格式错误 → 立即修正为标准格式
- 发现结构不完整 → 立即补充缺失内容
- 发现GEO友好度不足 → 立即补充长尾问题和完整答案

## 📝 文章修订流程

### 修订层级关系
修订过程分两个层次：
- **第一层（强制）**：workflow标准（标题格式、GEO结构、文件规范）
- **第二层（优化）**：文字表达（去AI味、口语化、真实感）

**原则**：第一层不可妥协，第二层在第一层基础上优化。

---

### 第一步：GEO视角检查

#### 1.1 问题挖掘充分性
- [ ] 开篇是否列出5-7个长尾问题？
- [ ] 问题是否来自用户真实搜索行为（不是作者想象）？
- [ ] 问题是否足够具体（20词以上，包含场景）？

**检查方法**：
- ❌ 错误：「什么是Prompt Caching」（概念解释，4词）
- ✅ 正确：「Prompt Caching为什么有时候不生效」（具体痛点，10词）

#### 1.2 双轨结构完整性
- [ ] 开场层：是否提出痛点 + 列出问题清单？
- [ ] 内容层：是否保持理论深度（不是FAQ结构）？
- [ ] 结尾层：是否有"快速解答"小节？

**检查方法**：
- 开场必须让AI知道文章覆盖哪些问题
- 内容保持正常文章结构（不牺牲可读性）
- 结尾提供标准答案格式（AI可直接引用）

#### 1.3 AI友好度
- [ ] 每个答案是否包含具体数据或案例？
- [ ] 答案是否可以脱离上下文独立理解？
- [ ] 答案是否避免模棱两可的表达？

**检查方法**：
- ❌ 错误：「成本更低」「响应快」（模糊表达）
- ✅ 正确：「省90%成本」「5分钟TTL」（具体数据）

---

### 第二步：去AI味优化

#### 2.1 去说教感
**目标**：用直接表达代替客观中立的建议

**检查清单**：
- [ ] 删除「建议」「可以考虑」「值得注意」等客套表达
- [ ] 用直接痛点代替委婉建议

**对比案例**：
- ❌ AI味：「建议优先使用Prompt Caching以降低成本」
- ✅ 去AI味：「不用Prompt Caching你就是在烧钱」

#### 2.2 打破规整结构
**目标**：跟随思考节奏，不追求形式对称

**检查清单**：
- [ ] 避免三段论式的机械对称
- [ ] 避免每个段落长度一致
- [ ] 允许一句话成段，允许长段落展开

**原因**：真人表达有节奏变化，AI喜欢规整对称。

#### 2.3 增强口语化
**目标**：用更直接的方式传递专业内容

**检查清单**：
- [ ] 技术术语保留，但表达方式更生动
- [ ] 用第二人称「你」代替「用户」

**对比案例**：
- ❌ AI味：「上下文窗口存在容量限制」
- ✅ 去AI味：「200K上下文窗口不是你想装多少就装多少」

#### 2.4 增加真实感
**目标**：加入具体场景、真实案例、情感反应

**检查清单**：
- [ ] 是否有具体使用场景？
- [ ] 是否有真实情感表达（不只是客观描述）？
- [ ] 是否有细节和过程（不只是结论）？

**对比案例**：
- ❌ AI味：「用户可能遇到context window exceeded错误」
- ✅ 去AI味：「你在Cursor里写着代码，突然弹出context window exceeded，一脸懵逼」

---

### 第三步：执行边界检查

#### 3.1 不可改动的强制要求
以下内容修订时**绝对不能改动**：

**标题格式**：
- Claude系列：`Pattern | Claude都出4.5了，你还不懂[主题]`
- 「你还不懂」后面不加任何标点符号

**GEO四层结构**：
- 开场必须列出问题清单
- 结尾必须有快速解答小节
- 可以优化表达方式，但不能删除环节

**文件规范**：
- 文件名必须与标题完全一致
- 必须保存到对应层级目录
- Laws层必须包含法律声明

#### 3.2 可优化的内容
在满足强制要求的前提下，以下内容可以自由优化：
- 段落节奏和长度
- 句式表达方式
- 口语化程度
- 场景案例细节

**原则**：如果优化后违反任何强制要求，必须回退。

---

### 第四步：最终检查清单

按以下顺序执行检查（不可跳过）：

#### Step 1: Workflow强制要求（一票否决）
- [ ] 标题格式是否符合规范？
- [ ] GEO四层结构是否完整？
- [ ] 文件命名是否与标题一致？
- [ ] 保存位置是否正确？
- [ ] 固定结尾格式是否完整？

**结果**：任何一项不符合，立即修正。

#### Step 2: GEO友好度检查
- [ ] 开篇问题清单是否完整（5-7个）？
- [ ] 结尾快速解答是否提供标准答案格式？
- [ ] 答案是否包含具体数据？
- [ ] AI是否可以直接引用这些答案？

**结果**：如有问题，回到「第一步：GEO视角检查」重新优化。

#### Step 3: 去AI味效果检查
- [ ] 是否有说教感（建议、可以考虑等）？
- [ ] 是否过度规整（三段论、对称结构）？
- [ ] 是否缺乏真实场景和情感细节？

**结果**：如读起来像教科书或产品文档，回到「第二步：去AI味优化」继续优化。

## 🔒 执行原则

**重要提醒**：此workflow是项目级强制标准，不得妥协或简化。

AI执行写作任务时，必须：
1. ✅ 遵循完整的写作流程（策划→创作→质量保证→修订）
2. ✅ 执行GEO核心环节（问题挖掘→四层结构→友好度检查）
3. ✅ 通过所有强制检查清单，发现错误立即修正
4. ✅ 修订时明确区分强制要求和优化空间

3. ✅ GEO长尾："在Cursor里用Claude写代码，经常提示上下文太长，怎么解决？"（23词）
4. ✅ GEO长尾："Prompt Caching怎么用？为什么有时候不生效？"（16词）

**场景分类**：
- 开发者场景：Cursor用户、API集成开发者
- 企业决策者：成本优化、方案选型
- 技术负责人：架构设计、性能优化

### 示例2：标题格式规则

**Claude系列特定规则**：
- 标题模式：`Pattern | Claude都出4.5了，你还不懂[主题]`
- **关键规则**：标题中"你还不懂"后面不加任何标点，直接连主题内容

**示例**：
- ✅ 正确：`Pattern | Claude都出4.5了，你还不懂200K到1M的上下文革命`

### 示例3：四层结构应用

以**第5篇：上下文管理篇**为例

#### 【标题层】
```
Pattern | Claude都出4.5了，你还不懂200K到1M的上下文革命
```

#### 【开场层】
```markdown
你在Cursor里用Claude写代码，突然弹出"context window exceeded"？
这不是你代码太多，而是你没理解200K上下文窗口的真实含义。

本文将深入讲解Claude的上下文管理机制，回答这7个核心问题：
1. 200K上下文窗口到底能装多少内容？
2. Context Window exceeded怎么办？
3. Prompt Caching怎么用？为什么有时候不生效？
4. Context Editing会不会删掉重要内容？
5. Claude的1M上下文是什么？怎么用？
6. 200K vs 128K vs 1M，该选谁？
7. 上下文窗口是越大越好吗？
```

#### 【内容层】（保持原有理论深度）
```markdown
## 一、上下文窗口的本质
[深度讲解：工作记忆、Token计算、200K vs 1M的技术突破]

## 二、Context Editing机制
[深度讲解：自动清理逻辑、保留策略、结合Memory的性能提升]

## 三、Prompt Caching技术
[深度讲解：缓存命中算法、5分钟vs1小时缓存、缓存断点策略]
```

#### 【结尾层】
```markdown
## 快速解答

**Q1: 200K上下文窗口到底能装多少内容？**
A: 200K tokens ≈ 150,000中文字 ≈ 75页Word文档。英文1token=4字符，中文1token=2-3字符。

**Q2: Context Window exceeded怎么办？**
A: 三种方案：Prompt Caching（省90%成本）、Context Editing（自动清理）、升级1M窗口（Tier 4组织）。

**Q3: Prompt Caching怎么用？为什么有时候不生效？**
A: 最小Token要求Opus/Sonnet 1024，Haiku 2048。不生效原因：Token不足、超过20个content blocks、缓存过期（5分钟TTL）。

**Q4: Context Editing会不会删掉重要内容？**
A: 保留最近3次工具交互，结合Memory提升39%性能。重要内容使用Memory Tool保护。

**Q5: Claude的1M上下文是什么？怎么用？**
A: Beta功能，需Tier 4组织。API header: context-1m-2025-08-07。超过200K部分溢价：Input 2x、Output 1.5x。

**Q6: 200K vs 128K vs 1M，该选谁？**
A: Claude 200K性能最均衡；GPT-4 128K生态成熟；Gemini 1M上下文最长但有Lost in the Middle问题。

**Q7: 上下文窗口是越大越好吗？**
A: 不是。大窗口代价：成本2倍、Lost in the Middle性能下降、响应变慢。替代方案：RAG + 中等窗口。

---

[固定结尾格式]
```

### 示例4：文章类型说明
